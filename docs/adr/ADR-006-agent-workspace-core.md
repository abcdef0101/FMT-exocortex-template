# ADR-006: интеграция DS-agent-workspace в ядро

**Статус:** принят
**Дата:** 2026-04-30
**АрхГейт:** пройден (⚠️ Эволюционируемость + Безопасность приняты)

## Контекст

Три роли (strategist, extractor, synchronizer) теперь auto-install.
С первого дня setup.sh они генерируют отчёты, находки, extraction-reports.

1. DS-agent-workspace был опциональным с момента создания, но критерии
   включения не были определены: пользователь не знает, в какой момент
   «уже пора» — 5 отчётов? 20? 50? Без порога решение откладывается.

2. Пользователь должен самостоятельно отслеживать количество файлов
   в DS-strategy/current/ и оценивать степень загрязнения workspace,
   чтобы принять решение о целесообразности подключения agent-workspace.
   Автоматического сигнала нет — система не говорит «пора».

3. Daily-report.sh содержал условную логику с fallback в DS-strategy/current/,
   которая при auto-install всех ролей больше не нужна: агенты гарантированно
   есть с первого дня.

## Решение

Интегрировать создание DS-agent-workspace в setup.sh как шаг 5.5.
Удалить fallback в daily-report.sh. Удалить setup/optional/.

## Альтернативы

- Флаг `--with-agent-workspace` — отвергнут: всегда нужен при auto-install трёх ролей
- Оставить опциональным — отвергнут: не решает проблему неопределённых критериев включения

## Последствия

- +1 обязательный репо, +1 шаг в setup.sh (против 6 ранее)
- Упрощение: убрана условная логика в daily-report.sh
- Пользователь больше не принимает решение «когда включать» — система решает сама
- Риски (приняты через АрхГейт): жёсткая зависимость от agent-workspace, потеря данных при сбое создания репо
