# ADR-009: IWE Behavior Validation Strategy

> **Status:** active
> **Created:** 2026-05-08
> **Replaces:** none
> **Superseded by:** none

---

## Context

IWE — это эмерджентное поведение AI-агента, который читает CLAUDE.md, следует протоколу ОРЗ, читает файлы workspace и пишет результат. Test coverage на 2026-05-08: 46 unit-тестов (100% pass), 5 E2E (setup/update/migration), 11 AI smoke-тестов (LLM-as-Judge). Пробел: нет способа **верифицировать что AI-агент выполняет протокол правильно** в условиях близких к production.

46 unit-тестов проверяют структуру файлов и синтаксис. 5 E2E — установку и обновление. 11 AI smoke-тестов — семантическое качество через LLM-judge, но их результаты недетерминированы (сегодня 7/8, завтра 6/8). **Отсутствует блокирующий CI gate «IWE работает как задумано».**

## Decision

Принять трёхслойную стратегию валидации поведения IWE:

```
Layer 1: Структурные ассерты (blocking CI gate)
  │  Проверяет детерминированные инварианты результата AI-процесса
  │  bash-assertions: grep, diff, git log
  │  100% deterministic, 0 cost, блокирует PR
  │
Layer 2: LLM-Judge (advisory, trend monitoring)
  │  Проверяет семантическое качество результата
  │  AI-оценщик через DeepSeek: 8 рубрик × порог
  │  Недетерминирован, сигнальный, не блокирует
  │
Layer 3: Canary replay (weekly health check)
  │  Перезапускает AI-процесс на исторических данных
  │  Сравнивает diff с фактическим результатом
  │  Обнаруживает деградацию модели/промпта
```

### Layer 1 — Assert-скрипты (blocking)

После `--run` (AI выполнил процесс) — детерминированная проверка структуры результата:

| Assert-скрипт | Что проверяет | Assertions |
|---------------|---------------|:----------:|
| `assert-day-close.sh` | DayPlan → итоги, таблица, commit | ~15 |
| `assert-week-close.sh` | WeekPlan → итоги, completion rate, content plan | ~12 |
| `assert-quick-close.sh` | WP Context → Осталось, →memory:, MEMORY sync | ~10 |
| `assert-capture-to-pack.sh` | KE маршрутизация: правило→CLAUDE.md, домен→Pack | ~10 |

### Layer 2 — LLM-Judge (advisory)

Уже реализовано: 11 AI smoke-тестов с рубриками (Phase 3 + R3). Результаты не блокируют CI, но тренд отслеживается.

### Layer 3 — Canary replay (weekly)

| Canary-скрипт | Что проверяет | Assertions |
|---------------|---------------|:----------:|
| `canary-day-close.sh` | Replay Day Close на копии workspace, diff | ~15 |
| `canary-wp-gate.sh` | WP Gate эмуляция: задача вне плана → STOP | ~8 |

## Alternatives considered

### Вариант A: Только LLM-Judge (отклонён)
LLM-оценщик для всех проверок. Недетерминирован. 7/8 сегодня, 6/8 завтра — ложные срабатывания в CI. Пригоден как сигнал, не как gate.

### Вариант B: Только E2E через golden image (отклонён)
Полный E2E в VM/контейнере каждый PR. Покрывает всё, но: дорого ($0.50-1.00/прогон), медленно (3-5 минут), не ловит регрессии в структуре результата.

### Вариант C: Трёхслойная (выбран)
Assert (блокирующий, дешёвый) + Judge (сигнальный, средняя стоимость) + Canary (replay, еженедельно). Покрывает все failure modes при минимальной стоимости.

## Consequences

### Positive
- CI получает блокирующий gate «IWE работает» — первый в истории проекта
- Canary-тест ловит деградацию модели/промпта до того как пользователь заметит
- Слои независимы — можно улучшать каждый отдельно

### Negative
- +6 тестовых файлов (~300 строк обслуживания)
- Canary-тест требует AI CLI ($0.50/прогон, еженедельно)
- Assert-скрипты хрупкие: изменение формата вывода → обновление grep-паттернов

### Mitigation
- Assert-паттерны параметризованы, не хардкод
- Canary запускается только weekly, не per-PR
- Failure в assert → re-run с --verbose для диагностики

## When to revisit

- При изменении формата DayPlan/WeekPlan/MEMORY.md → обновить grep-паттерны
- При смене AI-модели (deepseek → другая) → перекалибровать пороги judge
- При падении canary 3 недели подряд → пересмотреть Layer 3 подход
- При появлении детерминированного bash-инструмента для проверки семантики → заменить Layer 2
