## 1. Архитектура репозиториев

### Три типа репозиториев

| Тип | Подтип | Что содержит | Source-of-truth | Кто создаёт |
|-----|--------|-------------|-----------------|-------------|
| **Base** | Принципы | ZP, FPF, SPF | Да | Платформа |
| **Base** | Форматы | FMT-* (протоколы структуры) | Да (для формата) | Платформа |
| **Pack** | — | Доменные знания (вторые принципы) | Да | Пользователь |
| **DS** | instrument / governance / surface | Производные от Pack | Нет | Пользователь |

> Base = платформа выдаёт. Pack и DS = пользователь создаёт.
> Pack = вторые принципы. DS = третьи принципы.

### Fallback Chain

```
DS → Pack → Base (SPF → FPF → ZP)
```

Если знание не найдено в DS — ищем в Pack. Если нет в Pack — ищем в SPF. Если нет в SPF — ищем в FPF.

### Pack = source-of-truth

Pack содержит ТОЛЬКО доменное знание: методы, work products, failure modes, различения, SoTA-аннотации. **Нельзя:** планы, сроки, роли, код, дидактику.

### DS (Downstream) — три подтипа

| Подтип | Что содержит | Правило |
|--------|-------------|---------|
| `instrument` | Код, тесты, документация кода | Не менять source-of-truth |
| `governance` | Планы, реестры, решения, заметки | Не менять предметное знание |
| `surface` | Курсы, гайды, маршруты | Только потреблять знание, не создавать |

**DS/governance: DS-strategy** — личная база данных (ТОЛЬКО входы и выходы агентов). Строго нельзя: скрипты, код, промпты, launchd/systemd-конфиги.

**DS/instrument: именование** — `DS-{slug}`. Привязка к Pack через `source-of-truth: {ID}` в CLAUDE.md репо, не через имя.

### Repository-first режим

Никогда не отвечать из общих знаний без опоры на репозиторий:
1. Найти минимум 1-2 релевантных файла в репозитории
2. Собрать Context Pack
3. Только потом формировать ответ

Если контекст не найден → сообщить «в репозитории нет источника», предложить создать документ.

### Frontmatter (базовый)

```yaml
---
type: doc    # doc, spec, process, report
status: draft # draft, active, archived
created: YYYY-MM-DD
updated: YYYY-MM-DD
---
```

## 2. Memory: 3 слоя

### L1 — Платформа

CLAUDE.md §1-§7, навыки (skills/), протоколы (persistent-memory/protocol-*.md). Обновляется через `update.sh --check` / `--apply`. Это «неприкасаемый» слой — прямые правки запрещены (Extensions Gate).

### L2 — Staging

Правила на обкатке в `STAGING.md`. Работают → промоция при Week Close в L1 (через `template-sync.sh`). Статусы: `testing` → `validated` → `promoted`. Отклонённые → `rejected` (остаются в авторском L3).

### L3 — Авторское

CLAUDE.md §9, личные различения, конфигурация. Не затирается при `update.sh`. Включает:
- Блокирующие авторские правила
- Различения (AUTHOR-ONLY зона в `.claude/rules/distinctions.md`)
- Именование рабочих пространств
- Read-only репо

### Лимиты памяти

| Объект | Лимит |
|--------|-------|
| Файлов в memory/ | ≤11 |
| Справочников (fpf-reference, sota-reference, etc.) | ≤100 строк |
| Протоколов (protocol-*.md) | ≤150 строк |
| MEMORY.md | ≤100 строк |

### Temporal metadata

```yaml
valid_from: YYYY-MM-DD     # обязательно при создании
superseded_by: <файл>       # при устаревании
```

## 3. ОРЗ-фрактал (Open → Work → Close)

Три стадии, три масштаба. Пропуск Открытия = незапланированная работа. Пропуск Закрытия = незафиксированный результат.

| Масштаб | Открытие | Работа | Закрытие |
|---------|----------|--------|----------|
| **Сессия** | `protocol-open.md § Сессия` (WP Gate + Ritual, любое задание) | `protocol-work.md` (Capture-to-Pack, Self-correction) | Quick Close (4 шага: commit+push, WP Context, KE, MEMORY) |
| **День** | `/day-open` (сбор вчерашнего, формирование DayPlan) | Дневная работа (помидоры, слот 1 = self-dev) | `/day-close` (итоги, «Завтра начать с», MEMORY sync) |
| **Неделя** | Стратегическая сессия (понедельник) | Недельный цикл | `/week-close` (статистика, ротация уроков, аудит memory) |

### Блокирующие правила

1. **WP Gate:** ЛЮБОЕ задание → протокол Открытия → ДО начала работы.
2. **Push:** «заливай» / «запуши» → commit + push без дополнительных вопросов. Push ДО отчёта Закрытия.
3. **Close:** Триггер Закрытия → протокол Закрытия → выполнить.
4. **Чеклист-верификация (Haiku R23):** Quick Close и Day Close — sub-agent Haiku R23 (context isolation). Проверяет формальное соответствие чеклисту (все ли пункты закрыты, есть ли коммит, обновлён ли MEMORY.md), но не оценивает качество. Исключения: сессия ≤15 мин или без изменений файлов.
5. **Pull-on-Touch:** `git pull --rebase` при первом изменении в репо за сессию (не перед каждым коммитом).

### Протокол Работы

**Capture-to-Pack** — на каждом рубеже: есть ли знание для записи? Анонсировать: *«Capture: [что] → [куда]»*. Маршрутизация:
- Правило (1-3 строки) → `CLAUDE.md`
- Доменное знание → Pack
- Реализационное знание → DS docs/
- Урок → `memory/`

**Self-correction:** расхождение с заявленным → немедленно предложить фикс (файл, строка, что изменить).

## 4. Pre-action Gates

Восемь гейтов, срабатывающих перед началом работы в определённых условиях.

### Сводная таблица

| Gate | Триггер | Что проверяет | Блокирует? |
|------|---------|---------------|:----------:|
| **WP Gate** | Любое задание | РП в WeekPlan? Классификация: в плане / вне плана / срочная | **Да** |
| **ArchGate** | Архитектурное решение | 7 характеристик ЭМОГССБ, чеклист современности, порог ≥8 | **Да** |
| **IntegrationGate** | Новый инструмент/агент/система | (1) обещание → (2) сценарии → (3) роль → (4) реализация | **Да** |
| **Security Gate (B7.3)** | РП затрагивает PII (email, telegram_id, ЦД, tokens, user_events) | Ответить на §Б чеклист ArchGate. Логирование PII = блокер | **Да** |
| **Priority Gate** | РП ≥3h | К какому R{N} ведёт? | Нет |
| **SC Gate** | Пользовательский сценарий | Какое обещание (Service Clause в `08-service-clauses/`) затронуто? | Нет |
| **Repo-Touch Gate** | Первое содержательное действие в репо | Прочитать `<repo>/CLAUDE.md`. Если содержит «обязательно загружай» — загрузить указанные файлы ДО ответа | **Да** |
| **Pull-on-Touch** | Первое изменение в репо за сессию | `git pull --rebase` | **Да** |

### WP Gate — детали

**Проверка через `check-plan.md`:**
1. Задача есть в текущем WeekPlan? → продолжаем
2. Задачи нет в плане?
   - Срочная (дедлайн сегодня/завтра) → `wp-new` (быстрое создание РП)
   - Не срочная → спросить пользователя: добавить в план или отложить?
3. Задача противоречит плану? → сигнал к пересмотру приоритетов

### ArchGate — детали

**7 характеристик ЭМОГССБ** (v3 — conjunctive screening, профиль без агрегатного балла):

| Характеристика | Русский | Что оценивается |
|---------------|---------|-----------------|
| Evolvability (Э) | Эволюционируемость | Насколько легко изменить решение в будущем? |
| Modularity (М) | Модульность | Насколько независимы компоненты? |
| Observability (О) | Наблюдаемость | Насколько легко отслеживать состояние и ошибки? |
| Generativity (Г) | Генеративность | Насколько решение создаёт новые возможности? |
| Security (С) | Безопасность | Насколько защищены данные и доступ? |
| Speed (С) | Скорость | Насколько быстро работает? |
| Budget (Б) | Бюджет | Соответствует ли бюджетным ограничениям? |

**Чеклист современности:**
1. Context Engineering (SOTA.002)
2. DDD Strategic (SOTA.001)
3. Coupling Model (SOTA.011)

**Порог:** ≥8 из 10 возможных баллов = допустимо. АрхГейт — gate (допуск), не ranker (выбор лучшего).

### IntegrationGate — детали

**БЛОКИРУЮЩИЙ чеклист для новых инструментов/агентов/систем:**

1. **Обещание (Service Clause).** Что обещает? Кому? Какой результат? Записать в `08-service-clauses/`.
2. **Сценарии использования.** Кто запускает? Когда? Зачем? В каком контексте? Что делает с результатом? Минимум 3 сценария с разными потребителями. Приложить к Service Clause.
3. **Роль.** Какая роль (из DP.ROLE.001) отвечает? Создать новую если нет подходящей.
4. **Реализация.** Только после (1)-(3). Код, тесты, регистрация в hooks/config, smoke-test. Заголовок реализации должен содержать ссылку на Service Clause и Role.

**Исключения (IntegrationGate НЕ нужен):**
- Правка существующего инструмента без изменения его обещания
- Bugfix без изменения поведения снаружи
- Рефакторинг без функциональных изменений
- Экспериментальный скрипт на один запуск (но если запускается повторно — уже инструмент)

## 5. Стратегическая сессия

### 5.1 Session Prep (хедлесс, понедельник 04:00 UTC)

**Исполнитель:** R1 Стратег, сценарий `session-prep`
**Промпт:** `roles/strategist/prompts/session-prep.md`
**Триггер:** launchd/systemd-таймер `strategist.sh session-prep`

**Что происходит:**
1. Читает итоги прошлой недели из WeekPlan (секция `## Итоги W{N-1}`) — **не собирает** коммиты самостоятельно (это делает Week Review)
2. Обрабатывает inbox: `fleeting-notes.md`, все файлы в `inbox/`, QA bot report, WP context files
3. Проверяет Dissatisfactions (НЭП): какие можно закрыть, какие требуют обсуждения
4. Стратегическая сверка: читает `Strategy.md`, агрегирует `MAPSTRATEGIC.md` из всех репо, проверяет соответствие РП стратегии
5. Hub-and-Spoke: читает все `*/WORKPLAN.md` во всех репо, выявляет расхождения
6. Проверяет `Session Agenda.md` на нерегулярные блоки (ретро, архитектурные ревью и т.п.)
7. Строит контент-план на неделю (3-5 публикаций)
8. Архивация: предыдущий WeekPlan → `archive/week-plans/`, предыдущий DayPlan → `archive/day-plans/`
9. Архивация done WP context files, полная очистка `inbox/`
10. Создаёт черновик WeekPlan со статусом `draft`

**Выход:** `DS-strategy/current/WeekPlan W{N} YYYY-MM-DD.md` (`status: draft`)

### 5.2 Strategy Session (ручной, интерактивно с пользователем)

**Исполнитель:** R1 Стратег, сценарий `strategy-session`
**Промпт:** `roles/strategist/prompts/strategy-session.md`
**Precondition:** черновик WeekPlan должен существовать. Если нет → запустить session-prep.

**8 шагов:**

| # | Шаг | Содержание |
|---|-----|-----------|
| 1 | **Открытие** | Показывает агенду, ключевые вопросы, carry-over. Спрашивает: «Начинаем с повестки или срочных вопросов?» |
| 2 | **Ревью прошлой недели** | Разбор `## Итоги W{N-1}`: что получилось/нет, выводы |
| 3 | **Разбор Inbox** | Каждая заметка → пользователь решает: в план / capture / удалить / отложить. Решения применяются немедленно |
| 4 | **НЭП (Dissatisfactions)** | Закрыть решённые, обсудить новые. Обновление `Dissatisfactions.md` |
| 5 | **Стратегическая сверка** | Расхождения стратегии и текущих РП. Корректировка `Strategy.md` |
| 6 | **Формирование плана на неделю** | Пользователь корректирует РП: добавить/убрать, изменить бюджеты, приоритеты. Обновление таблицы РП |
| 7 | **Нерегулярные блоки** | Обработка ретро, архитектурных ревью, анализа документов |
| 8 | **Утверждение и синхронизация** | `status: draft → confirmed`. Обновление: `MEMORY.md` (таблица РП), `Strategy.md` (приоритеты, фазы), `MAPSTRATEGIC.md` (во всех репо), `WORKPLAN.md` (в целевых репо). Очистка `fleeting-notes.md` и `inbox/`. Commit + Push во всех затронутых репо |

**Выход:** `WeekPlan W{N}.md` (`status: confirmed`), обновлённые `Strategy.md`, `Dissatisfactions.md`, `MEMORY.md`, `WORKPLAN.md` (по всем репо), `MAPSTRATEGIC.md`, чистый `inbox/`.

## 7. Работа в течение дня

### Capture-to-Pack

На каждом рубеже работы — есть ли знание для записи? Анонсировать: *«Capture: [что] → [куда]»*.

Маршрутизация:
| Тип знания | Куда |
|-----------|------|
| Правило (1-3 строки) | `CLAUDE.md` |
| Доменное знание | Pack (конкретный файл предметной области) |
| Реализационное знание | DS docs/ |
| Урок | `memory/lessons_*.md` + строка в MEMORY.md |
| Идея контента | `drafts/draft-list.md` |

### Decision Capture

Только **пользовательские решения** (не решения агента) записываются в `DS-strategy/exocortex/decisions/decision-log-YYYY-MM.md` с cognitive weight (оценка когнитивной нагрузки решения). Новые решения — сверху, в секциях по датам.

### Quick Close (сессионный уровень)

**Триггеры:** «закрывай», «всё», «close».
**Длительность:** ~3 минуты (намеренно минимальный барьер).

**4 обязательных шага:**
1. **Commit + Push** — все изменения коммитятся и пушатся немедленно
2. **WP Context File** — обновить секцию «Осталось» (What's Left): что пробовали, что узнали, следующий шаг, контекст для продолжения
3. **KE (Knowledge Extraction)** — немедленная маршрутизация «что узнали»
4. **MEMORY.md** — обновить статус РП

**Верификация (Haiku R23):** проверяет — committed+pushed, контекст обновлён, KE смаршрутизирован, MEMORY обновлён, decision log точен.

### Self-correction

При обнаружении расхождения с заявленным поведением → немедленно предложить фикс (файл, строка, что изменить).

### Pomodoro и слоты

- **Слот 1 = self-development** — никогда не пропускать ради операционных задач
- **Цикл:** 25 мин работы / 5 мин перерыва
- **Длинный перерыв:** каждые 4 цикла
- **Break reminder:** после >50 мин непрерывной работы — напомнить о перерыве
- **Проверка плана:** между сессиями — «Всё ещё по плану дня?»

## 8. Note Review

**Исполнитель:** R1 Стратег, сценарий `note-review`
**Промпт:** `roles/strategist/prompts/note-review.md`
**Триггер:** ежедневно ~23:00 UTC (хедлесс) или ручной вызов

### Что делает

1. Читает `fleeting-notes.md`, находит все **bold** (новые) заметки
2. Читает контекст: Dissatisfactions, WeekPlan, Strategy, MEMORY, QA bot report
3. Классифицирует каждую заметку по 7 категориям:

| Категория | Действие |
|-----------|----------|
| **НЭП** | Предложение в конец `Dissatisfactions.md` |
| **Задача** | Предложение в конец WeekPlan / DayPlan |

### IWE Multiplier

```
multiplier = budget_closed_hours / wakatime_tracked_hours
```

- multiplier > 1.0: закрыто больше бюджета, чем трекалось времени (эффективность)
- multiplier < 1.0: времени потрачено больше, чем запланировано
- Таблица мультипликатора через `wakatime multiplier --budget <ч>`

### Роль ≠ Агент ≠ Инструмент (ключевое различение)

- **Роль** описывается независимо от исполнителя. Это функция (что делать), не система (чем делать)
- **Агент** — исполнитель роли. Может выбирать, КАК действовать (Grade 2+)
- **Инструмент** — не выбирает способ действия (Grade 1)
- **Агентская роль** (A.13) требует Grade 2+. **Функциональная роль** (A.2) — нет
- **Нотация FPF:** `Holder#Role:Context@Window`
- **Подготовка исполнителя:** program (скрипт) | prompt (LLM) | train (человек)

### Взаимодействие ролей

### ADI-цикл (`think`)

```
Abduction  → Формирование гипотез (что может объяснить ситуацию?)
Deduction  → Логическая проверка (следует ли решение из принципов?)
Induction  → Проверка данными (подтверждается ли гипотеза фактами?)
Audit      → Самопроверка (какие допущения сделаны? что упущено?)
Decide     → Решение с обоснованием
```

- `auditor [scenario]` — аудит завершён
- `extractor [scenario]` — экстракция завершена
- `strategist [scenario]` — стратегическая операция завершена

### VS Code интеграция

- **Compact dashboard** (20-30 строк) после Day Open: план, календарь, здоровье IWE, «Требует внимания»
- **Без Obsidian:** просмотр memory через VS Code

### WakaTime / IWE Multiplier

```
multiplier = budget_closed_hours / wakatime_tracked_hours
```

## 16. Полная недельная схема

```
НЕДЕЛЯ N-1 КОНЕЦ
│
├─ ВС→ПН 00:00  [WEEK REVIEW] хедлесс (R1)
│   └→ Итоги W{N-1} → секция в WeekPlan, клубный пост
│
├─ ПН    04:00  [SESSION PREP] хедлесс (R1)
│   └→ Читает итоги, inbox, стратегию, все WORKPLAN.md
│   └→ Создаёт черновик WeekPlan W{N} (status: draft)
│   └→ Архивация старых WeekPlan, DayPlans, WP-контекстов
│   └→ Очистка inbox/
│
├─ ПН (ручн)    [STRATEGY SESSION] интерактивно (R1 + R17-человек)
│   ├→ 1. Открытие: агенда, ключевые вопросы
│   ├→ 2. Ревью прошлой недели
│   ├→ 3. Разбор Inbox
│   ├→ 4. НЭП (Dissatisfactions)
│   ├→ 5. Стратегическая сверка
│   ├→ 6. Формирование плана на неделю
│   ├→ 7. Нерегулярные блоки
│   └→ 8. Утверждение: draft → confirmed + синхронизация
│       MEMORY.md, Strategy.md, WORKPLAN.md, MAPSTRATEGIC.md
│
├─ ВТ-ВС 04:00  [DAY OPEN] авто или /day-open (R1)
│   ├→ Вчерашний DayPlan (carry-over)
│   ├→ Коммиты за вчера, GitHub Issues
│   ├→ Календарь, Scout, IWE-здоровье
│   ├→ «Завтра начать с» → BLOCKING в план
│   ├→ Budget Spread, MEMORY.md cross-check
│   └→ DayPlan YYYY-MM-DD.md + compact dashboard
│
│   ДЕНЬ        [WORK: сессии ОРЗ]
│   ├→ Каждая сессия:
│   │   ├→ WP Gate → проверка задачи в плане
│   │   ├→ Работа: Capture-to-Pack на рубежах
│   │   │   ├→ Правило (1-3 строки) → CLAUDE.md
│   │   │   ├→ Доменное → Pack
│   │   │   ├→ Реализационное → DS docs/
│   │   │   └→ Урок → memory/
│   │   └→ Quick Close (4 шага):
│   │       Commit+Push, WP Context, KE, MEMORY.md
│   │
│   ├→ Decision Capture: только решения пользователя → decision-log
│   ├→ Slot 1 = self-development (никогда не пропускать)
│   └→ Pomodoro: 25/5, длинный перерыв каждые 4 цикла
│
│   ~23:00       [NOTE REVIEW] хедлесс (R1)
│   ├→ Классификация заметок (7 категорий)
│   ├→ Предложения в целевые документы
│   ├→ Архивация обработанных
│   └→ Очистка fleeting-notes.md
│
│   КОНЕЦ ДНЯ   [DAY CLOSE] (R1 + R23-верификатор)
│   ├→ Сбор коммитов, сопоставление с DayPlan
│   ├→ Обновление DayPlan, WeekPlan, WP-REGISTRY, MEMORY
│   ├→ Memory Drift Scan, Index Health, Lesson Hygiene
│   ├→ IWE Multiplier (budget / WakaTime)
│   ├→ Итоги: «Что сделано», «Что узнали», Praise
│   ├→ «Завтра начать с»: ВСЕ pending РП с действиями
│   ├→ Верификация (Haiku R23) → Commit + Push
│   └→ Compact dashboard
│
НЕДЕЛЯ N КОНЕЦ
│
└─ ВС (ручн)    [WEEK CLOSE] (R1 + R23)
    ├→ Ротация уроков (≤15 активных)
    ├→ Статистика недели: completion rate, коммиты, дни
    ├→ Инсайты, блокеры, carry-over
    ├→ Контент-план на следующую неделю
    ├→ MEMORY.md — свежая таблица РП
    ├→ ## Итоги W{N} в WeekPlan
    ├→ Клубный пост (R4 Автор)
    ├→ Memory audit: ≤11 файлов, лимиты строк
    ├→ /iwe-rules-review (DP.M.008)
    └→ Возврат к WEEK REVIEW для W{N} → ...
```

### Легенда к схеме

| Обозначение | Значение |
|------------|----------|
| `[HEADLESS]` | Автоматический запуск через launchd/systemd, без участия пользователя |
| `(ручн)` | Ручной запуск пользователем |
| `(R1 + R23)` | Участвуют роли R1 (Стратег) и R23 (Верификатор) |
| `→` | Производит артефакт или изменяет файл |
| `├→` / `└→` | Вложенные шаги |
