# Failure Injection Gate — Post-Remediation Verification

> Добавлен: 2026-05-07. Применяется после Phase 9 (Execute Remediation), перед Phase 10 (Fix-Review).

## Проблема

При аудите и последующем исправлении AI-агент склонен останавливаться на первом уровне валидации:

1. Увидел антипаттерн → исправил → прогнал happy path → галочка «готово»
2. НЕ проверил failure modes: что произойдёт если внешняя команда упадёт? пустой вывод? сеть недоступна?

Это приводит к silent false positives — скрипт-верификатор, который сообщает «OK» даже когда данные не получены.

## Правило

После ЛЮБОГО фикса (не важно — 1 файл или 10) применить **3 вопроса** к каждому изменённому блоку:

| Точка отказа | Вопрос |
|---|---|
| Внешняя команда в `$()` | Если exit ≠ 0 — скрипт узнает? Переменная будет пустой — что дальше? |
| Парсинг вывода | Если вывод malformed — ошибка видна или молча проглочена? |
| Сетевое/API обращение | Если API недоступен — скрипт упадёт с ясной ошибкой или пройдёт молча? |

## Процедура (блокирующая)

```
FOR each changed file:
  FOR each external command / parse / API call:
    1. Симулировать отказ (пустой ввод, malformed JSON, exit 1)
    2. Проследить цепочку до конца: что выведет скрипт? какой exit code?
    3. Если silent false positive → переоткрыть finding, создать новый
```

## Пример

Было:
```bash
BOARD_ISSUES=$(echo "$BOARD_RAW" | python3 -c '...' 2>&1 || echo "")
```

Вопросы:
1. Если python3 exit 1 — `$BOARD_ISSUES=""` — что дальше? **Ничего, скрипт продолжает с пустыми данными.**
2. Если JSON malformed — ошибка stderr → в $BOARD_ISSUES через 2>&1 — что дальше? **Grep не находит issue-номеров, скрипт молча проходит.**
3. Сеть — не применимо (python3 локальный).

Результат: silent false positive. **Добавить finding, исправить.**

Стало:
```bash
result=$(_py_parse "label" "$json" "$py_code")  # проверяет exit code + непустой вывод
```

## Новый антипаттерн: `command_substitution_without_exit_check`

**Сигнатура:** `var=$(cmd)` или `var=$(cmd 2>&1 || echo "")` — нигде не проверяется что cmd реально отработал. Данные могут быть пустыми, а скрипт продолжает исполнение.

**Почему вредит:** `set -e` не триггерит command substitution — скрипт молча продолжает с битыми данными. Особенно опасно в bash (в отличие от Python/JS, где исключение остановит выполнение).

**Исправление:**
1. Обернуть в helper: `_safe_capture() { local rc; result=$(cmd) || rc=$?; [ "$rc" -ne 0 ] && { echo "FAIL"; return 1; }; [ -z "$result" ] && { echo "FAIL: empty"; return 1; }; echo "$result"; }`
2. Всегда проверять результат на непустоту до использования
3. Stderr направлять в temp file, проверять отдельно
