# ADR-006: интеграция agent-output в ядро

**Статус:** принят (ревизия 2026-05-13)
**Дата:** 2026-04-30
**АрхГейт:** пройден (⚠️ Эволюционируемость + Обучаемость — приняты; ревизия снизила оба риска)

## Контекст

Три роли (strategist, extractor, synchronizer) теперь auto-install.
С первого дня setup.sh они генерируют отчёты, находки, extraction-reports.

1. DS-agent-workspace был опциональным с момента создания, но критерии
   включения не были определены: пользователь не знает, в какой момент
   «уже пора» — 5 отчётов? 20? 50? Без порога решение откладывается.

2. Пользователь должен самостоятельно отслеживать количество файлов
   в DS-strategy/current/ и оценивать степень загрязнения workspace,
   чтобы принять решение о целесообразности подключения agent-workspace.
   Автоматического сигнала нет — система не говорит «пора».

3. Daily-report.sh содержал условную логику с fallback в DS-strategy/current/,
   которая при auto-install всех ролей больше не нужна: агенты гарантированно
   есть с первого дня.

## Решение

Интегрировать создание директории `DS-strategy/agent-output/` в setup.sh
(через manifest) как обязательный элемент workspace. Удалить fallback
в daily-report.sh — писать всегда в `DS-strategy/agent-output/scheduler/reports/`.

## Альтернативы

- Флаг `--with-agent-workspace` — отвергнут: всегда нужен при auto-install трёх ролей
- Оставить опциональным — отвергнут: не решает проблему неопределённых критериев включения
- Отдельный git-репозиторий `DS-agent-workspace` — отвергнут: git избыточен. Единственный
  потребитель git-операций — daily-report.sh (push отчётов в remote). Остальные потребители
  (dt-collect.sh, tester, day-close.sh, strategist) только читают/пишут файлы. Remote backup
  отчётов синхронизатора не является критической функцией. Простая директория внутри
  основного репо покрывает все сценарии без дополнительного git remote в ментальной модели
  пользователя.
- Встроенная директория `DS-strategy/agent-output/` — **принят.** Изолирует agent output от
  human content (Context Engineering SOTA.002), не требует отдельного репозитория, снижает
  риски сбоя создания (mkdir вместо git repo create).

## Последствия

- +1 директория `DS-strategy/agent-output/` внутри workspace
- Упрощение: убрана условная логика в daily-report.sh
- Пользователь больше не принимает решение «когда включать» — система решает сама
- Риски (приняты через АрхГейт): fallback отсутствует, но сбой `mkdir` практически
  невозможен в отличие от сбоя `gh repo create`. При крахе директории данные теряются
  локально (вместе с основным репо) — риск эквивалентен потере самого DS-strategy.
- **Ревизия 2026-05-13:** форма workspace изменена с отдельного git-репозитория на
  встроенную директорию. Пункт «удалить setup/optional/» исключён — не относится
  к теме ADR. Рекомендована повторная оценка через 3 месяца.
